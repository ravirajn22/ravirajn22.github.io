Naming a local variable that normalises a function param

Date: 3 sept 2019

Consider the following sudo code sample 

      // Option 1
      function init(number userId) {
        userId = userIdOk(userId) ? userId : "someDefaultId"
        setupAccount(userId);
      }


      // Option 2
      function init(number userId) {
        String normalisedUserId = userIdOk(userId) ? userId : "someDefaultId"
        setupAccount(normalisedUserId);
      }

My aim is to check if the userId value is ok, if its not then assign a default id before calling the setupAccount.

In option 1, I reassign the value of the param userId if its not ok, where as in option 2, I create a new local variable called normalisedUserId to store it.

Pros and Cons
Generally programmars avoid mutating function param value for easy code readability (google it), But option 1, I am reassigning userId to another value (a form of mutation). In option 2, I create a new local variable instead of mutating the param. But the problem with option 2 is I have to give a new name to the variable userId, hence I have to come up with a new variable name called normalisedUserId.

After carefull consideration I went with option 1 for its brevity of having no new variables which might confuse the existing variables.

Note: In this case, the function is small and except the default check in no other place is the userId mutated. Different scenarios might require different solutions.
